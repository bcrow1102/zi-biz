# ZIMO BIZ 개발 기준

## 1. 서비스 정체성

ZIMO BIZ는 게임앱이 아니다.

부동산·분양 실무자를 위한 제작도구이자 홍보 플랫폼이다.

핵심 기능은 다음이다.

1. 디지털명함
2. 웹전단
3. 이미지 보정
4. 분양현장 홍보
5. 부동산 홍보 동영상
6. 중개업무 / 매물등록

게임이나 출석 기능은 나중에 붙더라도 DAU 보조 기능일 뿐, 중심 기능이 아니다.

---

## 2. 1차 베타 오픈 조건

아래 3개 기능이 구현되면 바로 베타 서비스한다.

1. 디지털명함
2. 웹전단
3. 이미지 보정

디지털명함만으로는 공개 베타를 열지 않는다.

단, 개발 우선순위는 현재 다음과 같다.

1. 디지털명함
2. 웹전단
3. 이미지 보정

다음 주중 1차 테스트 목표는 디지털명함과 웹전단 두 가지를 사람들이 실제로 써볼 수 있는 수준으로 만드는 것이다.

---

## 3. 이후 확장 순서

1. 분양현장 홍보
2. 부동산 홍보 동영상
3. 중개업무 / 매물등록

중개업무 / 매물등록은 지도, 주소, 검색, 필터, 이미지 저장, 알림, 정책, 검증, 비용 문제가 생길 수 있으므로 가장 마지막에 진행한다.

---

## 4. 저장 전략

초기 화면 구현과 편집 구조 안정화는 localStorage 또는 sessionStorage를 먼저 사용한다.

우선 안정화할 항목은 다음이다.

- 입력
- 미리보기
- 임시저장
- 다시 불러오기
- 초기화
- PDF 다운로드
- PPT 다운로드

Supabase는 데이터 구조가 안정된 뒤 연결한다.

단, 베타 서비스 전에는 다음 항목을 반드시 Supabase 기준으로 설계한다.

- 회원가입 / 로그인
- 회원별 작업물 저장
- 명함 공유 URL 생성
- 웹전단 다운로드 기록
- Gold 지급 / 사용 / 회수 장부
- 관리자 기능
- RLS 보안 정책

미저장 개인 입력값은 장기 localStorage에 남기지 않는다.

베타 또는 정식 서비스 전에는 미저장 초안은 sessionStorage 또는 메모리 상태로 관리하고, 로그아웃 또는 창 닫기 시 삭제되는 방향으로 정리한다.

의도적으로 저장하거나 발행한 작업물만 Supabase에 저장한다.

## 5. 베타 인증 정책

ZIMO BIZ 베타 서비스는 카카오 로그인과 이메일 승인번호 인증을 기본 로그인 방식으로 사용한다.

### 로그인 방식

- 카카오 로그인
- 이메일 승인번호 인증

### 비회원 사용 범위

비회원은 체험만 가능하다.

가능한 작업:

- 명함 입력 체험
- 전단 입력 체험
- 시안 선택 체험
- 배경 선택 체험
- 미리보기 확인
- 텍스트 수정 체험

불가능한 작업:

- 명함 공유 URL 생성
- 전단 PDF 다운로드
- 전단 PPT 다운로드
- 작업물 저장
- Gold 사용

### 회원 사용 범위

회원은 Gold가 있을 때 실제 결과물을 생성하거나 다운로드할 수 있다.

가능한 작업:

- 명함 공유 URL 생성
- 전단 PDF 다운로드
- 전단 PPT 다운로드
- 작업물 저장
- Gold 사용 내역 확인

### 이메일 인증 원칙

이메일 인증은 실명 인증이 아니라 이메일 소유 확인이다.

베타 단계에서는 한 사람이 여러 이메일로 가입할 수 있는 가능성을 허용한다.

정식 서비스에서는 필요 시 휴대폰 본인인증, 사업자 인증, 중개사 인증, 현장 인증 등을 추가한다.

### 카카오 로그인 원칙

카카오 로그인은 베타의 주 로그인 방식으로 사용한다.

같은 카카오 계정은 하나의 회원으로만 연결한다.

DB에는 카카오 고유 ID를 저장하고, 같은 카카오 고유 ID로 다시 로그인하면 새 계정을 만들지 않고 기존 회원으로 로그인 처리한다.

베타에서는 카카오 계정을 여러 개 만드는 문제는 큰 위험으로 보지 않는다.

실제 사용 권한은 Gold로 통제한다.

---

## 6. Gold 정책

ZIMO BIZ의 Gold는 실제 수익과 직결되는 사용권한이다.

전단 PDF/PPT 다운로드, 명함 공유 URL 생성 등 실제 결과물을 가져가는 행위는 Gold를 사용해야 한다.

### 기본 지급 정책

초기 정책은 다음을 기준으로 한다.

- 신규 가입 시 기본 Gold 지급
- 매월 기본 Gold 지급
- 출석 보상은 나중에 적용 가능
- 관리자 수동 지급 가능

단, 베타에서는 악용 방지를 위해 가입 Gold 자동 지급량을 조정하거나, 관리자 지급 방식으로 운영할 수 있다.

### Gold 사용처

Gold 사용처는 다음이다.

- 디지털명함 공유 URL 생성
- 웹전단 PDF 다운로드
- 웹전단 PPT 다운로드
- 추후 분양현장 홍보 상위 노출
- 추후 부동산 홍보 동영상 생성
- 추후 매물 홍보 기능

---

## 7. Gold 보안 원칙

ZIMO BIZ의 Gold는 실제 수익과 직결되는 사용권한이므로, 일반 회원이 직접 수정하거나 생성할 수 없어야 한다.

### 기본 원칙

1. 일반 회원은 자신의 Gold 잔액을 조회만 할 수 있다.
2. 일반 회원은 Gold를 직접 지급, 회수, 수정할 수 없다.
3. Gold 증감은 반드시 서버 함수 또는 Supabase RPC / Edge Function을 통해서만 처리한다.
4. 프론트엔드에서 Gold 값을 직접 업데이트하는 코드는 절대 만들지 않는다.
5. 모든 Gold 증감은 `gold_transactions` 장부에 기록한다.
6. 관리자 Gold 지급 / 회수 / 제한도 반드시 `admin_logs`에 기록한다.
7. 베타 단계에서 Gold 관리 권한을 가진 admin 계정은 발리 1명만 둔다.
8. 추후 운영자가 추가되더라도 Gold 지급 / 회수 권한은 별도 권한으로 분리한다.

### 일반 회원 권한

일반 회원은 다음 작업만 가능하다.

- 본인 Gold 잔액 조회
- 본인 Gold 사용 내역 조회
- Gold가 충분할 때 명함 공유 URL 생성 요청
- Gold가 충분할 때 전단 PDF/PPT 다운로드 요청

일반 회원은 다음 작업을 할 수 없다.

- Gold 잔액 직접 수정
- Gold 거래내역 직접 생성
- 다운로드 비용 직접 변경
- 가입 보너스 중복 수령
- 다른 회원 Gold 조회
- 다른 회원 Gold 수정

### 관리자 권한

베타 단계에서는 발리 관리자 계정만 다음 작업을 할 수 있다.

- 특정 회원에게 Gold 지급
- 특정 회원 Gold 회수
- 특정 회원 Gold 사용 제한
- 특정 회원 제한 해제
- 전체 Gold 발행량 조회
- 전체 Gold 사용량 조회
- 회원별 Gold 내역 조회
- 시안별 / 기능별 Gold 사용 통계 조회

### 필수 보안 구조

Gold 관련 테이블은 반드시 RLS를 적용한다.

- `profiles.gold_balance`는 일반 회원이 직접 update할 수 없다.
- `gold_transactions`는 일반 회원이 직접 insert / update / delete할 수 없다.
- Gold 차감은 `use_gold_for_action()` 같은 서버 함수에서만 처리한다.
- Gold 지급 / 회수는 `admin_adjust_gold()` 같은 관리자 전용 함수에서만 처리한다.
- 관리자 함수는 호출자가 `role = 'admin'`인지 반드시 확인한다.
- 모든 Gold 변경은 거래 장부와 관리자 로그에 남긴다.

### 절대 금지

다음 구조는 절대 만들지 않는다.

- 프론트엔드에서 `gold_balance` 직접 update
- 프론트엔드에서 `gold_transactions` 직접 insert
- 관리자 여부를 화면 코드에서만 검사
- 다운로드 비용을 프론트엔드 값만 믿고 처리
- 가입 보너스를 중복 방지 없이 지급
- 사용 제한 회원이 Gold 기능을 계속 사용할 수 있는 구조

## 8. 디지털명함 기준

디지털명함은 공유 링크 중심 기능이다.

사용자는 명함 정보를 입력하고 공유 URL을 생성한 뒤, 해당 링크를 카카오톡 또는 문자로 보낸다.

### 사용 흐름

1. 정보 입력
2. 시안 선택
3. 저장
4. 공유 URL 생성
5. 카카오톡 또는 문자로 링크 전송
6. 받은 사람은 공유 페이지에서 명함 확인

### 필수 기능

- 명함 정보 입력
- 시안 선택
- 저장
- 공유 URL 생성
- 공유 페이지 표시
- 카카오톡 / 문자에서 명함처럼 보이는 OG 미리보기

### Gold 사용

- 공유 URL 생성 시 Gold 차감
- 공유 URL 생성 기록을 남긴다.
- 같은 회원이 몇 개의 공유 URL을 생성했는지 관리자에서 확인할 수 있어야 한다.

## 9. 웹전단 기준

웹전단은 공유 링크가 필요 없다.

웹전단의 목적은 사용자가 앱에서 전단을 만든 뒤 PDF 또는 PPT 파일로 다운로드하고, 자기 컴퓨터에서 저장하거나 출력하는 것이다.

### 사용 흐름

1. 전단 정보 입력
2. 배경 선택
3. 시안 선택
4. 미리보기에서 텍스트 직접 수정
5. PDF 또는 PPT 다운로드
6. 사용자 컴퓨터에서 저장 또는 출력

### 필수 기능

- A4 텍스트 안착
- A6 텍스트 안착
- 배경 선택
- 시안 선택
- 텍스트 클릭 수정
- 초기화
- PDF 다운로드
- PPT 다운로드

### A4 / A6 기준

- A4 세로형 전단 제공
- A6 전단 제공
- A6는 A4 4등분 인쇄를 고려한다.
- A6는 상단 2칸 중심으로 편집 영역을 구성하고, 하단 2칸은 입력 선택 영역이 아님을 표시한다.

### 시안 기준

템플릿 이름은 설명형 이름을 쓰지 않는다.

사용하지 않을 표현:

- 심플
- 스탠다드
- 프리미엄
- 노랑
- 핑크
- 파랑

사용할 표현:

- 시안1
- 시안2
- 시안3

현재 베타 기준 웹전단은 시안1, 시안2 중심으로 정리한다.

시안3은 불필요하면 제거한다.

### 배경 기준

웹전단은 사용자가 마음대로 모든 이미지를 편집하는 방식보다, 완성된 배경을 선택하는 방식으로 간다.

초기 배경 계열은 다음을 우선한다.

- 아파트
- 오피스텔
- 일반

배경은 완성도 높은 제한된 선택지를 제공한다.

### 텍스트 기준

- 텍스트는 전부 수정 가능해야 한다.
- 텍스트는 미리보기에서 직접 클릭해 수정하는 방식을 우선한다.
- 텍스트 길이는 제한한다.
- 줄 수 제한을 둔다.
- 레이아웃이 깨지지 않도록 자유 편집을 과도하게 열지 않는다.
- 색상은 약 4개 프리셋으로 제한한다.
- 사용자가 더 세부 수정이 필요하면 PPT로 다운로드해서 직접 수정하게 한다.

### Gold 사용

- PDF 다운로드 시 Gold 차감
- PPT 다운로드 시 Gold 차감
- 다운로드 기록은 `usage_events` 또는 별도 다운로드 로그에 남긴다.
- 회원별 다운로드 수를 확인할 수 있어야 한다.
- 시안별 다운로드 수를 확인할 수 있어야 한다.

## 10. 이미지 보정 기준

이미지 보정은 1차 베타 오픈 조건에 포함된다.

단, 현재 개발 우선순위는 디지털명함과 웹전단을 먼저 서비스 가능한 수준으로 만드는 것이다.

이미지 보정은 이후 다음 기준으로 정리한다.

- 부동산 / 분양 실무에 필요한 이미지 보정 중심
- 복잡한 전문 편집툴이 아니라 간단한 실무 보정 중심
- 추후 Gold 사용 여부 검토

## 11. Supabase 주요 테이블 초안

Supabase 테이블은 확장성을 고려해 설계한다.

초기 예상 테이블은 다음이다.

### `profiles`

회원 기본 정보와 권한을 관리한다.

필요 항목 예시:

- `id`
- `user_id`
- `email`
- `kakao_id`
- `display_name`
- `role`
- `gold_balance`
- `is_restricted`
- `created_at`
- `updated_at`

### `digital_cards`

디지털명함 작업물과 공유 정보를 관리한다.

필요 항목 예시:

- `id`
- `user_id`
- `slug`
- `template_key`
- `data`
- `is_public`
- `created_at`
- `updated_at`

### `flyers`

웹전단 작업물 저장 정보를 관리한다.

필요 항목 예시:

- `id`
- `user_id`
- `size_mode`
- `template_key`
- `background_key`
- `data`
- `created_at`
- `updated_at`

### `gold_transactions`

Gold 지급 / 사용 / 회수 장부를 관리한다.

필요 항목 예시:

- `id`
- `user_id`
- `amount`
- `type`
- `reason`
- `related_item_type`
- `related_item_id`
- `created_by`
- `created_at`

### `usage_events`

기능 사용 기록을 관리한다.

필요 항목 예시:

- `id`
- `user_id`
- `event_type`
- `item_type`
- `template_key`
- `background_key`
- `cost_gold`
- `created_at`

### `admin_logs`

관리자 작업 기록을 관리한다.

필요 항목 예시:

- `id`
- `admin_user_id`
- `target_user_id`
- `action`
- `memo`
- `created_at`

## 12. 관리자 기능 기준

관리자는 서비스 운영과 Gold 관리를 위해 다음 정보를 확인할 수 있어야 한다.

### 회원 관리

- 전체 회원 목록
- 회원별 가입일

## 13. 개발 원칙

- 한 파일에 모든 기능을 몰아넣지 않는다.
- 기능별로 파일을 나눈다.
- main.js는 공통 화면 전환과 메뉴 중심으로 둔다.
- card.js는 디지털명함을 담당한다.
- flyer.js는 웹전단을 담당한다.
- image-tool.js는 이미지 보정을 담당한다.
- data.js는 localStorage / sessionStorage 저장과 불러오기를 담당한다.
- auth.js는 로그인 / 회원 / Gold 상태를 담당한다.
- Supabase 연결 후에는 DB 관련 코드를 별도 파일로 분리한다.
- Gold 관련 로직은 프론트엔드 직접 수정 방식으로 만들지 않는다.
- 관리자 권한 검사는 화면 코드가 아니라 DB 함수 / RLS / 서버 로직에서 처리한다.

## 15. 장기 확장 시 주의사항

향후 기능이 늘어나더라도 초기 구조를 과하게 복잡하게 만들지 않는다.

우선순위는 항상 다음 순서로 판단한다.

1. 실무자가 실제로 자주 쓰는가
2. 사용자가 쉽게 이해할 수 있는가
3. 수익 구조와 연결되는가
4. 개발 복잡도가 과도하지 않은가
5. 기존 핵심 기능을 흐리지 않는가

ZIMO BIZ는 부동산·분양 실무자를 위한 간단하고 실용적인 제작도구에서 출발한다.

기능을 많이 붙이는 것보다, 사용자가 실제로 쓰는 핵심 작업을 빠르고 쉽게 끝내게 하는 것이 우선이다.

## 로그인 후 입력값 보존 원칙

제작도구는 비회원도 입력과 미리보기 체험이 가능해야 한다.

비회원이 명함 공유 URL 생성, 전단 PDF/PPT 다운로드, 저장 등 로그인 필요한 작업을 클릭하면 로그인 팝업을 띄운다.

이때 사용자가 이미 입력한 값, 선택한 시안, 선택한 배경, 편집한 텍스트는 로그인 전후로 유지되어야 한다.

로그인 또는 회원가입이 완료된 뒤에도 사용자는 기존 작업 화면으로 돌아와야 하며, 입력값이 초기화되면 안 된다.

### 보존해야 할 항목

- 디지털명함 입력값
- 디지털명함 시안 선택값
- 웹전단 입력값
- 웹전단 A4/A6 선택 상태
- 웹전단 배경 선택값
- 웹전단 시안 선택값
- 웹전단 직접 수정한 텍스트
- 다운로드 또는 공유URL 생성 직전의 작업 상태

### 금지

- 로그인 완료 후 제작도구 첫 화면으로 강제 이동
- 로그인 완료 후 입력값 초기화
- 회원가입 완료 후 다시 처음부터 입력하게 만드는 구조
- 로그인 팝업을 닫았을 때 입력값이 사라지는 구조

### 구현 방향

로그인 전 임시 작업 상태는 sessionStorage 또는 메모리 상태에 보관한다.

로그인 완료 후 해당 임시 작업 상태를 복원하고, 사용자가 원래 누르려던 작업을 이어서 할 수 있게 한다.

예를 들어 비회원이 전단 PDF 다운로드를 눌렀다면:

1. 현재 전단 입력값을 임시 저장한다.
2. 로그인 팝업을 띄운다.
3. 로그인 완료 후 전단 화면으로 복귀한다.
4. 임시 저장된 전단 상태를 복원한다.
5. 사용자가 다시 PDF 다운로드를 진행할 수 있게 한다.
